# Bottleneck Analysis (병목 검증)

## 목적

섹션 4(`4.parallelization_proof/`)에서 확인한 다음 현상의 원인을 규명한다.

- **워커 동시 출발 등 병렬 동작 자체는 정상이지만, 복제 시간이 베이스라인보다 오히려 늘어남.**

본 섹션에서는 그 원인이
- (가) **병렬화는 잘 되어 있으나 어딘가에 직렬 병목이 존재**해서인지,
- (나) **실제로는 병렬화가 제대로 동작하지 않아서**인지

를 가려내고, 병목이라면 **어느 지점**인지를 좁힌다.

---

## 공통 실험 설정

### 빌드 / 부하 도구
- **빌드 모드: Release**
- **부하 도구: csql** (섹션 4 `csql_test`와 동일)

### 워크로드
- 테이블: tbl1 ~ tbl10 (10개)
- 각 테이블에 **10만 건 Insert**를 **단일 트랜잭션**으로 입력
- 모든 실험에서 동일

### 처리 흐름 (대상 시스템)
1. **Reader**가 로그 파일을 읽으며 replication 로그를 감지 → 단위 **Item** 리스트 생성
2. **Commit 로그 감지 시 트랜잭션 단위로 워커에 할당** (병렬화 진입점)
3. 트랜잭션을 할당받은 **Worker**가 수행:
   - Item을 읽음
   - 각 Item에 해당하는 레코드 데이터 조회
   - **repl obj 생성**
   - **flush (200개 단위 배치)**
   - 모든 Item 소모 후 **Commit**

### "활성화" 의 정의
- 워커의 **`repl obj 생성` + `flush` 수행 여부**로 활성화/비활성화를 구분.
- **활성화된 워커**: 위 처리 흐름을 그대로 수행 → 슬레이브 서버에 데이터 저장 요청을 보냄.
- **비활성화된 워커**: repl obj 생성·flush를 수행하지 않음 → 슬레이브 서버에 저장 요청을 보내지 않음.

## 1차 측정: csql 부하 테스트

측정값 상세:
- [`1.csql_test/worker_1.md`](1.csql_test/worker_1.md) — 활성화 워커 1개 측정 (활성 워커는 `worker_id = 2`)
- [`1.csql_test/worker_10.md`](1.csql_test/worker_10.md) — 활성화 워커 10개 측정

### 데이터 정리

| 항목 | worker_1 (활성 1, worker_id=2) | worker_10 (활성 10) |
|------|--------------------------------|---------------------|
| 마스터 복제 시간 | 80.40초 | 57.55초 |
| 슬레이브 복제 시간 | **8.99초** | 59.17초 |
| 전체 복제 시간 | 85.07초 | 113.31초 |
| 마스터 평균 Duration | 79.03초 | 55.96초 |
| 슬레이브 평균 Duration | **1.48초** | 47.55초 |

> 판정 기준 적용 시 `worker_10 ≫ worker_1` 명확히 성립 (슬레이브 기준 약 **6.6배** 차이).

### 분석

#### 병목 위치: **슬레이브 측 쓰기/flush 경합이 주범**

활성 워커 1 → 10으로 늘릴 때 슬레이브 복제 시간이 **8.99초 → 59.17초로 약 6.6배 폭증**.

- 워크로드(테이블 수·데이터량·트랜잭션 수)와 마스터 측 동작은 동일하게 유지.
- 변한 것은 **슬레이브 쪽에서 동시에 flush/저장 요청을 받는 워커 수**뿐.
- 즉, 다수 워커가 동시에 슬레이브에 쓰기를 보낼 때 어딘가에서 **직렬화/경합**이 발생.
- → 4번 섹션의 *"병렬은 보이지만 빨라지지 않는다"* 의 원인이 **슬레이브 측 동시 쓰기 경합**으로 좁혀짐.

#### tbl2 이상치 (worker_1, 슬레이브 4.74초)

- worker_1에서 활성 워커는 worker_id=2 한 개 → 슬레이브 쓰기가 사실상 순차 처리.
- 슬레이브 종료 시각으로 보면 tbl2가 가장 마지막(09:14:34.839)이며, Duration 4.74초로 다른 테이블 대비 유난히 김.
- 단일 활성 워커가 처리 큐 끝에서 tbl2 처리에 유난히 오래 머문 케이스로 추정.
- 단발 이상치인지 데이터 특성인지는 재측정으로 확인 필요.

### 1차 결론

> **병렬화는 정상 동작하지만, 슬레이브 측 동시 쓰기 경합이 직렬 병목으로 작용한다.**

4번 섹션의 *"병렬은 보이지만 빨라지지 않는다"* 가설을 **정량으로 입증**.

## 2차 측정: perf 프로파일링

### 측정 목적 (왜 perf인가)

1차 결론에 따라 **슬레이브 측 경합 지점이 flush로 좁혀졌고, 이를 구체적으로 식별**하기 위해 `perf`를 수행한다.

- 대상: 슬레이브 측 applier 워커 프로세스 (동시 쓰기 부하 상태)
- 환경: worker_10 케이스(= 동시 쓰기 부하가 가장 큰 상태)에서 수집, 워커 1개 케이스와 대비
- 목적: 어느 함수/락/IO 지점에서 시간이 소비되는지를 콜스택 기반으로 좁히기

측정값 상세:
- [`2.applier_perf_test/worker_1.md`](2.applier_perf_test/worker_1.md) — 활성 워커 함수 호출 트리
- [`2.applier_perf_test/worker_10.md`](2.applier_perf_test/worker_10.md) — 워커 10개 각각의 함수 누적 시간

### 결과 — 함수 호출 트리 비교

워커 한 명이 동일한 ~17,000회 force 호출을 처리한 누적 시간:

```
                                            1_worker tbl2  →  10_worker 평균    비율
                                            ───────────────   ────────────────   ──────
  ─ la_flush_repl_items                          3.335 s   →     45.81 s         × 13.7
    │ self  (add_repl_obj 등 준비)               0.149 s   →      0.211 s        ×  1.4
    └─ locator_repl_mflush_force                 3.186 s   →     45.60 s         × 14.3
       │ self                                    0.035 s   →      0.047 s        ×  1.3
       └─ locator_repl_force                     3.151 s   →     45.55 s         × 14.5
          │ self                                 0.043 s   →      0.059 s        ×  1.4
          └─ net_client_request_…copyarea        3.108 s   →     45.49 s         × 14.6
             │ self  ★ 응답 대기                 2.940 s   →     45.286 s        × 15.4
             └─ css_send_req_to_server_2_data    0.169 s   →      0.205 s        ×  1.2
```

서버 측 — per-call 평균 (워커당 누적이 아니므로 분리):

```
                                             1_worker     →   10_worker 평균      비율
                                            ───────────────   ────────────────   ──────
  ─ slocator_repl_force                          130 µs   →   2,595 µs           × 19.9
    │ self                                        16 µs   →      18 µs           ×  1.1
    └─ xlocator_repl_force  ★ 경합 핫스팟        114 µs   →   2,577 µs           × 22.5
```

시간이 어디로 갔는가 — 워커당 누적 막대 (1칸 = 5 %):

```
  1_worker tbl2 (3.335 s 100%)
    [██████████████████░░]  recv self = 응답 대기   2.940 s  88.2 %
    [█░░░░░░░░░░░░░░░░░░░]  send                    0.169 s   5.1 %
    [█░░░░░░░░░░░░░░░░░░░]  la_flush + mflush + force self   0.227 s   6.7 %

  10_worker 평균 (45.81 s 100%)
    [████████████████████]  recv self = 응답 대기  45.286 s  98.9 %
    [░░░░░░░░░░░░░░░░░░░░]  send                    0.205 s   0.4 %
    [░░░░░░░░░░░░░░░░░░░░]  la_flush + mflush + force self   0.317 s   0.7 %
```

### 분석

- **클라이언트 측 시간의 거의 전부가 `recv` self = 슬레이브 응답 대기** — worker_1에서도 88.2%, worker_10에서는 **98.9%**.
- send는 두 케이스 모두 < 1초로 무시 가능 → 네트워크 송신 부하 아님.
- 클라이언트 측 자체 처리(la_flush/mflush/force의 self)도 1초 미만으로 무시 가능 → 클라이언트 측 CPU 부하 아님.
- 서버 측 per-call 비교에서 **`xlocator_repl_force` 가 114 µs → 2,577 µs로 약 22.5배** 증가 → **슬레이브 서버 내부의 경합 핫스팟**이 명확.

### 2차 결론

> 1차 결론에서 좁혀진 "슬레이브 측 동시 쓰기 경합"의 실체는, 구체적으로 **`xlocator_repl_force` 경로의 동시 호출 경합**이다.

- 클라이언트 측은 보낸 뒤 응답을 기다리는 데 시간을 거의 다 쓰고 있고,
- 서버 측에서는 `xlocator_repl_force` per-call 시간이 워커 수 증가에 따라 약 22배 증가함.
- → 다음 분석은 슬레이브 서버 측 `xlocator_repl_force` 내부로 들어가야 함.

## 3차 측정: 슬레이브 서버 측 perf 프로파일링

### 측정 목적 (왜 server perf인가)

2차 결론에서 `xlocator_repl_force` 경로의 동시 호출 경합이 핫스팟임이 드러났다. 그러나 그 내부에서 **시간이 어디로 가는지**(어떤 락? 어떤 sleep? 어떤 페이지 작업?)는 클라이언트(applier) 측 perf만으로는 식별할 수 없다.

따라서 슬레이브 **서버 프로세스**에 대해 perf를 추가 수집해, 동시 쓰기 부하 상태에서 서버 내부의 어느 함수/락에서 시간이 소비되는지를 확인한다.

### 선결 검증: 서버가 정말 병렬로 동작하는가

병목 원인을 분석하기 전에, **슬레이브 서버가 다수 워커의 요청을 실제로 병렬 처리하고 있는지부터 확인**해야 한다.

- 만약 서버 내부에서 요청이 단일 스레드/큐로 **직렬화**되어 있다면, 앞서 관측된 응답 지연은 "경합"이 아니라 **설계상 직렬 처리**가 원인이며, 의심 지점 분석 자체가 무의미해진다.
- 따라서 이 선결 검증이 통과한 후에야 아래 의심 지점 검증이 유효하다.

#### 검증 결과 (요약)

> 상세: [`3.server_perf_test/parallelism_verification.md`](3.server_perf_test/parallelism_verification.md)

- `slocator_repl_force` 를 호출한 서버 TID 분포 확인 결과:
  - 총 호출 수 **170,001** (= 10 트랜잭션 × 17,000, applier 측과 일치 → 누락 없음)
  - 유니크 TID **19개** (기대치 10의 약 2배)
  - 호출이 여러 TID에 분산 (17k → 2k까지 불균등)

→ **서버는 직렬화가 아니라 병렬로 동작 중, 선결 검증 통과**. 따라서 아래 의심 지점 분석은 유효.

> **서버 측 매핑 구조 (회의로 확정)**: 클라이언트 연결은 워커가 아닌 **"코어"(논리 단위)** 에 매핑되며, **1 코어 = 6 워커 (실제 스레드)** 풀. 클라이언트 리퀘스트는 매핑된 코어 내 가용 워커에 할당됨 → **1 클라이언트 : 1 코어 : 6 워커 풀** 구조. (10 클라이언트 → 코어 10개 → 이론 최대 60 TID, 실제 19 TID 관측은 코어당 평균 ~1.9 워커 사용)
>
> **다만 작업 분배가 고르지 않음** (최상위 TID 17k vs 최하위 2k, 약 8배 격차) — 코어 내부 워커 할당이 요청 도착 타이밍·워커 상태에 의존하는 우연성 때문으로 해석되며, **별도 분석이 필요한 항목**으로 분리해 둠.

### 의심 지점

`xlocator_repl_force` 내부 경합 원인 후보를 우선순위 순으로 정리:

| 순위 | 의심 지점 | CPU 상태 | 비고 |
|------|-----------|----------|------|
| 1순위 | WAL 버퍼가 가득 찼을 때 워커가 **1 ms 씩 잠드는 코드** (H1) | CPU **on** | 동시 쓰기로 WAL 압이 올라가면 빈번한 sleep으로 누적 지연 가능 |
| 2순위 | 모든 로그 기록이 통과하는 **단일 mutex (`prior_lsa_mutex`)** (H2) | CPU **off** | 다수 워커가 같은 mutex에서 직렬 대기 |
| 3순위 | **인덱스 페이지가 분할될 때** (H3) | CPU **off** | page split 시 latch 보유 시간 증가 → 다른 워커 대기 |

### 측정 방법

- 대상: 슬레이브 데이터베이스 서버 프로세스 (`$SLAVE_PID`)
- 환경: worker_10 케이스 (= 동시 쓰기 부하가 가장 큰 상태)
- 명령: `perf record -F 1999 --call-graph=fp -p $SLAVE_PID -- sleep 30`
- 후처리: `slocator_repl_force` / `xlocator_repl_force` 스택만 필터링

### 결과 — 가설 매칭 (요약)

> 상세 콜체인 + 해석: [`3.server_perf_test/worker_10.md`](3.server_perf_test/worker_10.md)

| ID | 매칭 함수 inclusive % | 트리 등장 |
|----|----------------------|-----------|
| H1 `thread_sleep` | 0.00% | ✗ (비활성) |
| H2 `__lll_lock_wait` / `prior_lsa_next_record_internal` / `futex_wait` | **21.21% (on-CPU), 97.21% (off-CPU)** | ✓ 다수 위치 ★H2 |
| H3 `pgbuf_block_bcb` / `pgbuf_lock_page` | 0.04% | ✗ (비활성) |

**핵심 관찰:**
- **H2 단독 지목** — heap·btree·tran stats 모든 로깅 경로가 동일 `prior_lsa_mutex`로 수렴.
- **락 자체보다 "락 안의 LZ4 압축"이 본질** — `prior_lsa_alloc_and_copy_crumbs → log_zip → LZ4_compress` (11.03%)가 critical section 내부에서 실행되어 락 보유 시간을 늘림. per-call 22배 폭증의 메커니즘.
- 부수 신호로 `lock_object` (10.92%, lock-free 해시맵) → H2 해결 후 2차 병목 후보로만 기록.

### 3차 결론

> **병목의 실체는 `prior_lsa_mutex` contention (H2)**. 본질은 "락 자체"가 아니라 **락 안에서 수행되는 LZ4 압축이 락 보유 시간을 늘리는 것**.
>
> H1·H3는 본 워크로드에서 비활성으로 배제.
