# 슬레이브 서버 — 병렬화 선결 검증 (worker_10)

병목 원인 분석(H1/H2/H3 의심 지점 검증) 전에, **슬레이브 서버가 다수 워커의 요청을 실제로 병렬 처리하는지** 를 먼저 확인.

- 활성화 워커: 10 / 비활성화 워커: 0
- 빌드 모드: Release / 부하 도구: csql
- 워크로드: tbl1 ~ tbl10, 각 10만 건 Insert / 단일 트랜잭션
- 측정 대상: 슬레이브 데이터베이스 서버 프로세스
- 관측 대상 함수: `slocator_repl_force`

---

## 가설

- 활성 워커 = 10 → 서버 측에서 `slocator_repl_force` 를 수행하는 **TID도 10개** 일 것으로 기대.
- (각 클라이언트 워커당 서버 측 1 핸들러 스레드라는 단순 모델)

## 관찰

1. **TID 개수가 기대(10) 의 약 두 배(19)** — 클라이언트:서버 = 1:1 핸들러 모델이 아님.
2. **최상위 TID 3892375 = 정확히 17,000 calls** = 정확히 **한 트랜잭션 전체**.
   - 즉, 적어도 한 트랜잭션은 하나의 TID에서 끝까지 처리됨.
3. **나머지 트랜잭션은 여러 TID에 분산** — 호출이 17k → 2k까지 매우 불균등하게 분포.
4. **TID 번호 군집이 두 덩어리**:
   - `3892xxx` 군: 6 TIDs, 합계 67,719 calls (39.8%)
   - `3893xxx` 군: 13 TIDs, 합계 102,282 calls (60.2%)
   - 두 군의 PID 간격(~1,000)으로 보아, 워크로드 진행 중 **추가 스레드가 동적으로 생성**되었을 가능성.

## 해석 (가설 → 회의로 확정)

가설 단계에서 후보로 거론된 시나리오:

- (a) 스레드 풀 + 동적 확장
- (b) 워커당 보조 스레드 동반
- (c) 트랜잭션 핸들 이관(thread migration)

→ 실제 구조는 아래 "서버 측 병렬 처리 구조" 참조 (회의 + 디버그 로그로 확정).

## 서버 측 병렬 처리 구조 (회의 결과)

> 출처: 예찬님과의 회의 + 이해 바탕으로 디버그 로그에 정보 추가하여 확인.

### 핵심 매핑 구조

- 클라이언트 **연결**은 워커가 아닌 **"코어"(논리 단위)** 에 매핑됨.
- **1 코어 = 6 워커 (실제 스레드)** 로 묶임.
- 클라이언트의 **리퀘스트**는 매핑된 코어 내부의 워커 중 하나에 할당됨.
- → **하나의 클라이언트 요청이 코어 내 여러 워커에 걸쳐 처리될 수 있음**.

### 구조도

```
    Client (applier worker)              Server
    ────────────────────────       ─────────────────────────────

    ┌──────────┐                   ┌──── Core 1 ────────────┐
    │ Client A │── 연결 (1:1) ───►│  W1  W2  W3  W4  W5  W6 │
    └──────────┘                   └─────────────────────────┘
         │                          (코어 내 워커 6개 풀)
         │  request 1 ─────────────► 가용 워커에 할당
         │  request 2 ─────────────►
         │  request 3 ─────────────►

    ┌──────────┐                   ┌──── Core 2 ────────────┐
    │ Client B │── 연결 (1:1) ───►│  W1  W2  W3  W4  W5  W6 │
    └──────────┘                   └─────────────────────────┘

         ...                            ...   (Core 10까지)
```

### 워커 할당 정책

| 상황 | 동작 |
|------|------|
| 코어에 신규 작업 + 워커 A 가용 | → 워커 A에 할당 |
| 워커 A 작업 중 + 다음 작업 도착 | → 워커 B에 할당 |
| 워커 A 완료 + 다음 작업 도착 | → 워커 A 재할당 가능성 있음 |

→ 한 클라이언트의 요청들이 **코어 내 여러 워커에 분산**되며, 분산 정도는 요청 도착 시점과 워커 상태의 우연성에 좌우됨.

### 본 측정(19 TID)과의 연결

- 활성 클라이언트 워커 10개 → 매핑된 코어 10개 → **이론적 최대 60 TID** (= 10 × 6).
- 실제 관측 **19 TID** ≈ 코어당 평균 약 **1.9 워커**가 사용됨.
- **분배 불균형(17k ↔ 2k)** 의 원인: 코어 내부 워커 할당이 우연성(요청 도착 타이밍 + 워커 상태)에 의존 → 한 코어 내에서 하나의 워커가 우세하게 잡거나, 여러 워커로 잘게 나뉠 수 있음.

이로써 앞서 "1:N 매핑"으로 표현했던 부분이 구체화됨: **1 클라이언트 : 1 코어 : 6 워커 풀** 구조.

## 검증 결론

- **서버는 직렬화가 아니라 병렬로 동작 중**: 호출이 19개 TID에 분산되어 있음 → **선결 검증 통과**.
  - 따라서 `xlocator_repl_force` 내부의 의심 지점(H1/H2/H3) 분석은 **유효**.
- **그러나 작업 분배가 고르지 않음** — 최상위 TID 17k vs 최하위 2k로 **약 8배 격차**.
  - 일부 TID는 한 트랜잭션을 끝까지 처리하고, 다른 TID들은 자잘한 조각으로 나뉘어 처리됨.
  - 분배 불균형의 원인(스케줄링 정책 / 트랜잭션 이관 / 풀 동작 등)은 **별도 분석이 필요**한 항목으로 분리.
- 추가로 **클라이언트:서버 스레드 매핑이 1:1이 아니라 1:N (또는 풀 기반)** 이라는 점도 기록.
  - 본 병목과는 별개 이슈지만, *워커당 성능* 지표 해석 시 영향 가능.

## 별도 분석 필요 항목

- **작업 분배 불균형 분석**
  - 어느 TID가 어떤 트랜잭션/테이블을 얼마나 처리했는지 매핑
  - 분배가 시간순으로 어떻게 변화했는지 (동적 풀 확장 시점, 핸들 이관 시점)
  - 분배 불균형이 전체 처리 시간(꼬리 형성)에 미치는 영향 정량화
