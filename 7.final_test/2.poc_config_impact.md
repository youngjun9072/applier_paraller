# Report 2 — POC branch: Config Tuning 효과

## 한 줄 요약

POC branch의 parallel applier에서는 master config tuning이 slave 복제 시간을 **약 2배 단축**하고, 워커당 처리 속도도 약 40% 개선한다.

## 비교 대상

| 실험 | Config |
|---|---|
| **7.poc_baseline** | default (addvoldb 100G 만) |
| **6.poc_buf5g_dwb0** | dwb=0, data_buffer=5G, log_buffer=5G, log_volume_size=1G, checkpoint_interval=30min, addvoldb temp |

> `6.poc_buf5g_dwb0` 의 config는 `2.dev_tuned` 와 **동일**합니다. 동일 tuning 하에서 build branch만 다른 비교 페어이며, develop vs POC applier 효과를 깨끗하게 비교할 수 있습니다.

## 시간 비교

> **지표 정의**
> - `Slave Elapsed`: 첫 slave apply → 마지막 slave apply 까지의 wall-clock
> - `Slave/worker` (평균): `Slave Sum / 10` (테이블당 평균 처리 시간, Slave Sum = 모든 테이블 slave 시간 합)
> - `eff.par`: `Slave Sum / Slave Elapsed` (평균 동시 활성 워커 수, 이론 최대 = 10)

### Insert

| 실험 | 총 (Slave Elapsed) | 평균 (Slave/worker) | eff.par |
|---|---:|---:|---:|
| 7.poc_baseline | 68.5s | 27.3s | 3.99 |
| 6.poc_buf5g_dwb0 | **25.9s** | **16.1s** | **6.23** |
| 절감 | **−62%** | **−41%** | +56% |

### Update

| 실험 | 총 (Slave Elapsed) | 평균 (Slave/worker) | eff.par |
|---|---:|---:|---:|
| 7.poc_baseline | 48.2s | 28.0s | 5.80 |
| 6.poc_buf5g_dwb0 | **32.6s** | **16.3s** | 5.01 |
| 절감 | **−32%** | **−42%** | −14% |

## 분석

- 두 실험 모두 **PARALLEL** slave
- **워커당 처리 속도**가 약 40% 빨라짐 (27.3 → 16.1s, 28.0 → 16.3s)
  → buffer / dwb 같은 config 효과는 슬레이브 측에서도 동일하게 발현
- **총 slave 시간** Insert −62%, Update −32%
  - 차이가 큰 이유: 워커가 빨라지면서 동시성이 자연스럽게 올라감
- Update의 eff.par는 5.80 → 5.01로 소폭 하락했지만, 워커가 빨라져서 전체 elapsed는 줄어듦

## 결론

**Parallel applier 환경에서 config tuning은 강력한 추가 효과.**
1.dev_config_impact 의 sequential 결과와 정반대 — POC에서는 master 측 tuning이 slave 측 워커 속도까지 끌어올린다.

→ data_buffer / dwb 중 어떤 요인이 얼마나 기여했는지는 [Report 3](3.poc_parallelism_factors.md) 참조.
