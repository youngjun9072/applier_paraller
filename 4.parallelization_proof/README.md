# Parallelization Proof (applylogdb 병렬화 증명)

병렬화 적용 이후의 동작과 성능을 정량 측정하여 다음 두 가지를 검증한다.

1. 실제로 **병렬 수행**되고 있는가?
2. 그 결과 **성능 향상**이 발생하는가?

---

## 목적

- 섹션 3(`3.dev_log/`)에서 구현한 병렬화 구조가 의도대로 동작하는지 측정값으로 확인.
- 섹션 1(`1.develop_profile/`)의 베이스라인(직렬, 슬레이브 복제 시간 29.42초)과 비교하여 성능 향상 여부 판단.

## 측정 시나리오

- 대상: tbl1 ~ tbl10 (10개 테이블) 동시 부하
- 측정값: 테이블별 시작/종료 시각
- 도출 지표:
  - **마스터 복제 시간** = 마스터 최초 Start → 마스터 최종 End
  - **슬레이브 복제 시간** = 슬레이브 최초 Start → 슬레이브 최종 End
  - **전체 복제 시간** = 마스터 최초 Start → 슬레이브 최종 End

## 베이스라인 대비 (Release 기준)

| 항목 | Baseline (직렬, 섹션 1) | 병렬화 버전 | 변화 |
|------|--------------------------|--------------|------|
| 병렬 동작 | 없음 (순차 시작) | 있음 (동시 시작) | 구조 달성 |
| 슬레이브 복제 시간 | 29.42초 | 49.53초 | **약 1.68배 증가** |

## 관찰

1. **병렬 수행은 정상 동작 확인**
   - 마스터/슬레이브 모두 다수 테이블의 시작 시각이 거의 동시(수십 ms 내)에 찍힘.
   - 베이스라인의 직렬 시작 패턴(한 테이블이 끝나야 다음 테이블 시작)과 명확히 대비됨.
   - → 구조적 병렬화는 달성됨.

2. **그러나 복제 시간은 오히려 증가**
   - 슬레이브 복제 시간: 29.42초(직렬) → 49.53초(병렬), 약 1.68배 증가.
   - 가장 느린 테이블이 전체 종료 시점을 결정하며 꼬리(tail)가 길게 형성됨.
   - → **"병렬은 보이지만 빨라지지 않았다."**

3. **추론: 어딘가에 직렬 병목이 존재**
   - 워커들이 동시 출발해도 어디선가 직렬화되어 처리량이 늘지 않는 신호.
   - 가능한 후보: 공유 자원(락/큐/커밋 지점) 경합, 단일 디스패치 경로, 마스터 측 로그 생성/전달 처리량 한계 등.
